# VCT 项目 — 三人分工文档

## 核心理念：先搭地基 → 一个模板 → 按模板复制

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3
地基        模板页       按模板复制    收尾
┌──────┐   ┌──────┐   ┌──────────┐   ┌──────┐
│ A 做  │→  │ B 做  │→  │ C + A 做 │→  │ 三人  │
│ 地基  │   │ 模板页 │   │ 其余页面  │   │ 收尾  │
└──────┘   └──────┘   └──────────┘   └──────┘
```

**为什么这样分？**

- 没必要三个人同时理解数据管道和模型细节 — Person A 搞定后，B/C 只管调接口
- 先做一个完整页面，比写文档更能说明"页面该怎么写"
- 后面两个人只需要复制模板页的代码结构，改数据源和图表即可

---

## Phase 0：地基 — Person A（~2 天）

**目标是：让 B/C 能调用的接口就绪，不需要理解 ML 细节**

### Task 0-1: 环境搭建
- venv + requirements.txt（pandas, scikit-learn, xgboost, streamlit, plotly 等）
- 确认目录结构完整

### Task 0-2: 数据下载
- Kaggle 数据集下载到 `data/raw/`
- 记录各 CSV 的字段含义

### Task 0-3: data_pipeline 最小可用版
- 读 raw CSV → 类型转换 → 合并成主表
- 产出 `data/processed/`
- 只做必要清洗，不追求完美

### Task 0-4: 特征工程 + 模型训练
- `features.py`：构建 core features（队伍近期 form、地图胜负、ACS 移动平均）
- `models.py`：训练一个可用的 XGBoost，Walk-forward CV 验证
- 模型保存到 `models/`

### Task 0-5: predictor.py 基础版
- 接口只需要三个函数。这是 B/C 唯一需要关心的接口：

```python
# B/C 只需要调用这三个函数，不用理解内部
predict_match(team1, team2)       → {"team1_win_prob": 0.62, "team2_win_prob": 0.38}
predict_bp(team1, team2)          → {"ban_prob": {...}, "pick_prob": {...}}
predict_bo3_score(team1, team2)   → {"2-0": 0.25, "2-1": 0.30, "1-2": 0.25, "0-2": 0.20}
```

## Phase 1：模板页面 — Person B（~1.5 天）

**目标是：一个完整的页面，作为 C 的「代码样板」**

### Task 1-1: Streamlit 主入口 + 公共组件
- `app/main.py`：侧边栏导航 + 统一布局
- `app/components/`：卡片、选择器、图表包装器等

### Task 1-2: Match Predictor 页面（模板页）

这个页面要刻意写成「样板房」的效果 — 关键位置加注释：

```python
# ===== TEMPLATE: 数据加载区 =====
# 后续页面在这段替换为自己的数据源
df = load_match_data()
teams = get_team_list(df)

# ===== TEMPLATE: 计算区 =====
# 在这段调用 predictor 或数据处理逻辑
result = predictor.predict_match(team1, team2)

# ===== TEMPLATE: 渲染区 =====
# 使用 components/ 中的公共组件绘图
show_win_prob_gauge(result)
show_bp_matrix(result)
```

**模板页包含的「可复制要素」：**
| 要素 | 说明 |
|------|------|
| 选择器 → 数据加载 → 计算 → 渲染 三段式结构 | 所有页面统一 |
| 公共组件调用方式 | `from components import ...` |
| Plotly 图表配置风格 | 配色、字体、布局 |
| 错误 / 空状态处理 | `if df.empty:` 的处理写法 |
| Streamlit session_state 用法 | 缓存数据、避免重复加载 |

## Phase 2：按模板扩展 — Person C + Person A（~2 天）

**目标是：复制模板页结构，换数据即可**

Person B 的模板页完成后，C 和 A（地基做完后空闲）各领一个页面开发：

### Person C 负责

| 页面 | 主要区别 | 难度 |
|------|----------|------|
| Team Dashboard | 不调 predictor，直接从 processed 数据聚合展示 | ★★ |
| Masters London | 调 predictor，重点是 bracket 可视化 | ★★★ |

### Person A（地基完成后）负责

| 页面 | 主要区别 | 难度 |
|------|----------|------|
| Player Analyzer | 不调 predictor，以选手维度聚合展示 | ★★ |

**开发方式：**
1. 打开 `app/pages/1_match_predictor.py` 作为参考
2. 复制文件，改名
3. 替换数据加载区（自己的数据源）
4. 替换计算区（自己的计算逻辑）
5. 复用渲染区代码 + 组件

## 接口契约（Person A 对 B/C 的承诺）

`predictor.py` 提供以下接口，B/C 不关心内部实现：

```python
from src.predictor import predict_match, predict_bp, predict_bo3_score
from src.data_pipeline import load_processed_data

# 数据
df = load_processed_data()  # → DataFrame

# 预测
predict_match("G2 Esports", "EDward Gaming")
# → {"team1_win_prob": 0.58, "team2_win_prob": 0.42, "predicted_winner": "G2 Esports"}

predict_bp("G2 Esports", "EDward Gaming")
# → {"team1_ban_probs": {"Ascent": 0.3, ...}, "team1_pick_probs": {"Haven": 0.4, ...}, ...}

predict_bo3_score("G2 Esports", "EDward Gaming")
# → {"2-0": 0.20, "2-1": 0.38, "1-2": 0.28, "0-2": 0.14}
```

## 代码规范（模板页会体现）

- **三段式页面结构**：数据加载 → 计算 → 渲染（用 `# ====` 分隔）
- **公共组件优先**：先看 `app/components/` 有没有现成的
- **数据路径**：统一引用 `DATA_DIR = "data/"`，不硬编码
- **依赖 predictor**：只调三个接口函数，不直接加载模型
- **session_state**：页面切换时不重复加载数据
