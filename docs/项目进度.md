### Phase 0-3: data_pipeline 最小可用版（✅ 2026-05-31）

**实现内容：**
- `src/data_pipeline.py`：读取 raw CSV → 类型转换 → 合并主表 → 输出 parquet

**关键实现决策：**
- scores.csv（match 级）与 overview.csv（player-match-map 级）关联：先将 overview 聚合到 match-team 级别（跨选手取均值），再按 Team A / Team B 分拆合并到 scores
- 百分比字段（KAST、HS%）自动去除 `%` 后缀再转浮点数（原始 CSV 中为 `"78%"` 格式）
- 输出格式：parquet（比 CSV 快 10x，自带列压缩）

**产出：**
- `data/processed/match_dataset.parquet` — 12478 场比赛 × 34 列（含 team_a/team_b 统计量和 target 标签）
- `data/processed/draft_dataset.parquet` — 19164 条 BP 记录

**已知问题：**
- team_a_rating / team_b_rating 空值率约 25%（2023+ 部分 match-team 在 overview 中无对应记录），后续在训练时填充

### Phase 0-4: 特征工程 + 模型训练（✅ 2026-05-31）

**实现内容：**
- `src/features.py`：队伍 form（滚动窗口 5/10/20）、统计差值（diff_kd/diff_acs 等 7 项）、历史对战 H2H
- `src/models.py`：XGBoost + Logistic Regression，Walk-forward CV（4 folds：2023→2026）
- `src/evaluation.py`：Brier / LogLoss / AUC / F1 / McNemar / Calibration ECE

**结果：**
| 模型 | Brier | AUC | Accuracy |
|------|-------|-----|----------|
| XGBoost | 0.065 | 0.973 | 0.910 |
| Logistic | 0.067 | 0.969 | 0.899 |

**产出：**
- `data/features/match_features.parquet` — 12478 × 55（含 21 个特征列）
- `models/xgb_model.pkl` + `models/imputer.pkl`
- `models/feature_importance.csv` / `models/cv_results.csv`

**注意：** diff_kd（重要性 53%）、diff_acs（16%）等特征来自 overview 表（比赛结束后统计），用作赛前预测存在数据泄漏。predictor.py 分两条路径：
- 赛中模式（已有部分实时数据）：用全部特征
- 赛前模式（仅团队名称）：只用 form + H2H 特征

### Phase 0-5: predictor.py（✅ 2026-05-31）

**实现内容：**
- `src/predictor.py`：三个预测接口 + 模型/数据缓存的懒加载

**关键实现决策：**
- **赛前 vs 赛中双路径**：赛前 diff_* 特征置 NaN，由 imputer 填充训练均值；赛中传实时统计计算真实差值
- **form 查询**：从 match_features 中取队伍最近一场的 form_5/10/20 值，跨 Team A/Team B 位置归一化
- **H2H**：按 team1 视角返回 h2h_ratio，若仅对手方有记录则 `1 - ratio`
- **BP（地图）**：基于 draft_dataset 中队伍历史 ban/pick 频率，返回 top N 及全局统计
- **BO3 比分**：`P(2-0) = p²`, `P(2-1) = 2p²(1-p)` 链式乘法，假设每图独立同分布

**产出：**
- `src/predictor.py` — 三个函数：predict_match / predict_bp / predict_bo3_score

**接口：**
```python
predict_match(team1, team2, stats_a=None, stats_b=None)  # → {"team1_win_prob": ..., "mode": "pre_match"}
predict_bp(team1, team2, n_top=5)                         # → {"team1_bans": [...], "global_ban_rate": {...}}
predict_bo3_score(team1, team2)                           # → {"2-0": ..., "2-1": ..., "1-2": ..., "0-2": ...}
```

## 技术方案

### 开发策略

先搭地基（数据→特征→模型→预测接口），再做 Match Predictor 完整页面作为**模板页面**，其余两人照着模板复制开发剩余页面。

### 三人分工

```
Phase 0           Phase 1           Phase 2            Phase 3
Person A(地基) →  Person B(模板页) → Person C+A(复制) → 三人(收尾)
```

### 模型方案

- **Match Predictor**: XGBoost + Elo Rating + Logistic Regression 集成
- **BP 预测**: 条件概率 P(ban X | team 选了 Y)
- **BO3 比分**: P(2-0) = P(map1 win) × P(map2 win) 链式乘法
- **特征**: ACS, KAST, ADR, HS%, 历史对战, 近期 form（滚动窗口 5/10/20 场）, 统计差值

### 接口约定

predictor.py 只暴露三个函数给前端：

```python
predict_match(team1, team2)       → {"team1_win_prob": ..., ...}
predict_bp(team1, team2)          → {"ban_prob": ..., "pick_prob": ...}
predict_bo3_score(team1, team2)   → {"2-0": ..., "2-1": ..., "1-2": ..., "0-2": ...}
```

### 代码约定

- **页面三段式**：数据加载 → 计算 → 渲染，`# ====` 分隔
- **先看 `app/components/`** 有无现成组件
- **数据路径**：统一 `DATA_DIR` 常量
- **模型接口**：每个模型实现 `train()` 和 `predict()`
- **类型注解**：src/ 下的函数需 type hints

### 评估指标

| 指标 | 用途 |
|------|------|
| Brier Score / Log Loss | 概率校准质量 |
| ROC-AUC | 区分能力 |
| Accuracy / F1 | 分类准确率 |
| McNemar's Test | 模型间显著性检验 |
| Walk-forward CV | 时序验证（防止数据泄漏） |
